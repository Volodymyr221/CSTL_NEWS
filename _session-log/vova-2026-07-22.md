## Задача A — новини: чому «Україна та світ» показували плашку «Читати повністю» + фікс

- **Скрін Вови (IMG_3476/3477):** статті «Україна та світ» досі показували плашку «Джерело надає лише анонс через RSS — Читати повністю». «Нам такого не треба! Нам потрібен повний текст. Скажи, чому так сталося, якщо ми це виправляли?»
- **Причина:** статті спаршені РАНІШЕ (перші налаштування парсера), ДО появи логіки дотягування повного тексту → зберігся лише короткий анонс (<500 симв.), без прапорця `fullText`. Парсер дедуплікує за URL → вже бачені посилання пропускає → старі «анонси» ніколи не перепарсовувались. Клієнтська умова плашки: `!exclusive && sourceUrl && !fullText && text<600` → усі збіглись.
- **Фікс (системний, `scripts/parse_rss.py`, коміт `c231071d` прямо в main):**
  - `rehydrate_short_articles(existing_articles)`: на КОЖНОМУ прогоні доганяє повний текст для збережених «анонсів» на місці (той самий `fetch_full_article`/`fetch_rayon_article` що й для нових). Лічильник спроб `_fullTries` (макс 3) + бюджет 40 запитів/прогін — щоб не смикати нескінченно джерела без повного тексту.
  - offline-бекфіл прапорця `fullText=True` для 44 старих статей що вже мали текст ≥500 симв., але без прапорця → плашка на них зникла одразу (без мережі), закомічено в `data/articles.json`.
  - save в `main()` тепер спрацьовує і коли нових статей нема, але re-hydrate щось змінив (`if new_articles or rehydrated`).
- **Обмеження середовища:** пісочниця не має зовнішнього інтернету (egress policy) → мережеві дотяги роблю НЕ я, а CI (workflow rss-parser). Вова запускає workflow вручну (workflow_dispatch через API дає 403).
- **Статус:** parser-фікс + offline-бекфіл задеплоєно (прямий коміт у main). Решта ~28 коротких статей дотягнуться на наступному прогоні парсера в CI.
