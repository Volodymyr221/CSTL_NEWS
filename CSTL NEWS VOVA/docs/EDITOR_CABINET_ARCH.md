## 0. Головне відкриття, що визначає все

Аналіз коду виявив **дві несумісні площини** й **6 пасток**, через які «наївний» кабінет (писати новини в Supabase і зливати зі стрічкою на фронті) — крихкий:

1. **id-колізія.** git-id рахуються `max(id)+1` лише по `articles.json` (2380-2975), нічого не знають про Supabase. `openArticle` шукає `find(a.id === id)` строгим `===`. Дубль id → відкриється не та стаття; UUID → `Number()`=NaN → клік мертвий.
2. **Кеш «читається раз».** `allArticles` у пам'яті заповнюється один раз за життя сторінки, без realtime. Свіжа публікація Алли не з'явиться в живій сесії.
3. **Асиметрія Service Worker.** `data/*.json` — network-first із кешем (офлайн = старе). Supabase — без кешу (офлайн = зникає). Злита стрічка стає неповною/непослідовною.
4. **`curated.json` — фантом.** Задокументований, але НЕ реалізований. Готового злиття двох джерел для новин у коді немає.
5. **Несумісність полів.** `posts` (Дошка) не мають `geo/ts/category/content` → пряме злиття ламає фільтри й сортування стрічки.
6. **Дві площини публікації.** Новини = git + CI (браузер НЕ вміє комітити). Дошка = Supabase + RLS. Edge-функції Supabase теж НЕ комітять у git.

**Висновок:** не змішувати джерела на фронті. Потрібна інша вісь.

## 1. Рішення: «Supabase редагує — Git публікує» (єдиний конвеєр)

**Принцип:** фронтенд читає ОДНЕ джерело — git `articles.json`, як і зараз. Supabase — це виключно шар РЕДАГУВАННЯ. Усі 6 пасток зникають, бо фронт не змінюється.

```
Алла (браузер)                  CI (GitHub Action)              Читач
─────────────                   ──────────────────              ──────
пише чернетку  ──► Supabase ──► синк-джоб (service_role):       фронт читає
cms_articles      (RLS-гейт)    бере status='ready',            лише git
статус draft                    нормалізує у git-схему,         articles.json
   │                            жене через publish_queue →      (кеш/офлайн/
клік «Публікувати»              articles.json (git-id, дедуп,   id — усе як є)
статус → ready                  деплой), статус → published
```

**Чому стабільно:**
- Фронтенд НЕ чіпаємо — жодної з 6 пасток (id/кеш/SW/поля/realtime) не виникає.
- Уся надійність (id, дедуп, ліміти, баланс, деплой) — **перевикористана** з наявного `publish_queue.py`.
- Контент Алли й контент авто-агента течуть через ОДНУ машинерію публікації → уніфікація.
- Безпека — справжня (RLS + service_role лише в CI), не ховання кнопок.

## 1b. Єдина консоль керування (адмінка + кабінет + метрики) — рішення Роми

Не окрема сторінка кабінету, а **ОДНА консоль** (еволюція `admin.html`) з розділами, які показуються за дозволами (RBAC). В одному місці: перевірка чернеток статей, модерація Дошки, оголошення, метрики. Збігається із затвердженою архітектурою «одна спільна панель + permissions».

**Розділи консолі (кожен — за окремим дозволом):**
| Розділ | Що робить | Дозвіл | Джерело даних |
|---|---|---|---|
| 📝 Статті (чернетки) | перегляд/редагування/публікація чернеток Алли | `news.create`/`news.publish` | Supabase `cms_articles` → git |
| 🎪 Події «Шо в селі» | те саме для подій | `events.*` | `cms_articles type=event` |
| 📋 Модерація Дошки | approve/reject постів мешканців | `board.moderate` | Supabase `posts` (наживо, RLS) |
| 🏛️ Оголошення | офіційні оголошення сільради | `board.announce` | Supabase `announcements` |
| 📊 Метрики | DAU, пости/тиждень, свіжість стрічки, здоров'я | `metrics.view` | `HEALTH.md` + лічильники Supabase |
| 👥 Люди й ролі | додати редактора/адміна, розставити дозволи | `users.manage` | `editor_users`/`admins` |
| ⚙️ Налаштування | пароль/профіль | усі | Supabase auth |

**Матриця ролей (приклад):**
| Роль | Статті | Події | Дошка | Оголош. | Метрики | Люди |
|---|---|---|---|---|---|---|
| Супер-адмін (Вова/Рома) | ✅+публ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Редактор (Алла) | ✅+публ | ✅ | — | — | 👁️ | — |
| Модератор (майбутнє) | — | — | ✅ | — | 👁️ | — |

**Ключове:** видимість розділу в UI — лише зручність. СПРАВЖНІЙ гейт кожної дії — RLS/edge за відповідним дозволом. Одна перевірка `has_perm(розділ)` і в UI (показати), і в БД (дозволити) — щоб не було діри.

**Метрики — фундамент уже є:** `scripts/health_check.py` пише `HEALTH.md` (свіжість стрічки, баланс, к-сть за розділами). Консоль показує його + живі лічильники Supabase. Розвивається окремо.

## 2. Безпека (RLS, за best practices 2026)

- **RBAC через security-definer `has_editor_perm()`** (не рекурсивний запит у політиці; роль `authenticated`; RLS на всіх таблицях). Best practice підтверджено дослідженням.
- **Окрема таблиця `editor_users`** (грануляція create/publish) — НЕ додаємо Аллу в `admins` (це дало б повний адмін без грануляції).
- **`is_admin()` НЕ чіпаємо** (тримається на `auth.email()` — на ньому вхід Вови). Супер-адміни автоматично мають і права редактора.
- **`cms_articles` публічно НЕ читається** (нема anon-select) — фронт бере git, редактори бачать через `authenticated`+perm, CI — через service_role.
- **Google-вхід додаємо ПОРУЧ** з email (email збігається → працює), не мігруємо примусово. Не чіпаємо `phase_b_rls` (заблокує всіх).
- **Публікація (draft→ready)** гейтиться в БД: `WITH CHECK (status<>'ready' OR has_editor_perm('publish'))` — редактор без `publish` не проштовхне в ефір.

## 3. Перевикористання (нічого не винаходимо)

- **`publish_queue.py`** — синк-джоб живить його ж (id/дедуп/деплой безкоштовно).
- **Модель походження картинок** (`imageType` source/illustration/none + credit), яку я щойно зробив — Алла отримує її одразу; її фото → `source`, ілюстрації позначені.
- **Патерн fallback** `renderBoardBlock` — для новин не потрібен (фронт лишається git-only), але доступний.
- **Прапорець-джерела в масиві** (`auto:true` у подіях) — зразок, як CI мітить редакторські статті.
- **Edge-функція** `send-chat-push` (verify_jwt + перевірка прав + service_role) — зразок, якщо згодом захочемо МИТТЄВУ публікацію (repository_dispatch) замість опитування.

## 4. Форвард-сумісність (закладено в дизайн)

- **`type` (news/event)** у `cms_articles` → той самий кабінет обслужить «Шо в селі» (події Алли). Один движок.
- **Агент-review (майбутнє):** черга авто-агента теж може лягати в `cms_articles` як чернетки на перегляд Алли → одна поверхня review, той самий синк публікує.
- **Планування:** статус `scheduled` + `publish_at`; синк-джоб поважає час.

## 6. Головне рішення на узгодження

**Синхронізація Supabase→git:**
- **A) Опитування (polling):** CI-джоб кожні ~10-15 хв бере `ready`-статті. Просто, без нових секретів-у-браузері, невелика затримка публікації. **Рекомендовано для старту.**
- **B) Миттєво (edge-функція + repository_dispatch):** публікація за секунди, але потрібен PAT в edge-функції. Додати згодом за потреби.

Рекомендація: **A зараз, B — коли захочемо миттєвість.**
