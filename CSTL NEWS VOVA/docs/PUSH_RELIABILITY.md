## 1. У чому проблема (коротко)

Після фічі «Збережені рейси» кнопка-дзвіночок 🔔 у списку збережених **обіцяє нагадування, яких насправді не буде** у кількох випадках. Це гірше за відсутність фічі: користувач думає що рейс відстежується і не прийде вчасно, бо понадіявся на сповіщення.

«Дзвіночок увімкнено» ≠ «push реально прийде». Треба зробити їх **тотожними** (чесними) і **надійними**.

## 2. Що ПЕРЕВІРЕНО в коді (факти, не здогади)

Файл: `src/tabs/buses.js`.

### 2.1. 🔴 Push працює ЛИШЕ для сьогоднішніх рейсів
- `subscribeToPush()` (≈рядок 107) починається з:
  ```js
  if (trackDate !== getTodayISO()) return;   // ← виходить мовчки
  ```
- `unsubscribeFromPush()` (≈рядок 171) — той самий guard.
- Підписка створюється **тільки** в момент збереження рейсу (track-add) і при перемиканні дзвіночка.
- **Наслідок:** зберігаєш рейс **на завтра/майбутнє** → дзвіночок у списку «увімкнено» → але **підписка на push не створюється взагалі** → нагадування **НЕ прийде**. І немає механізму який підпише цей рейс, коли настане його день.

### 2.2. 🟡 Стан дзвіночка не відображає реальний стан підписки
- Прапорець `notify` (true/false) у записі трекінгу — це **лише локальне бажання** користувача, не факт що push активний.
- Якщо `subscribeToPush()` тихо вийшов (немає дозволу на сповіщення, не iOS-PWA, обірвалась мережа, впав запит) — `notify` все одно `true`, дзвіночок «увімкнено». **Брехня без зворотного зв'язку.**

### 2.3. 🟡 Залежність від крихкого пайплайну
- Push тримається на: VAPID-ключах, таблиці `push_subscriptions`, Edge Function `send-bus-push`, розкладі `pg_cron`.
- Історія тихих збоїв (з SESSION_STATE / комітів): ротація VAPID лишала старий ключ у функції → 401 → `sent:0`; `dep_time null` → підписка не матчилась → сповіщення не йшли. Тобто пайплайн ламається тихо.

## 4. Підводні камені Web Push на iOS-PWA (дослідити в інеті перед рішенням)

> Це найкапризніша частина. Перед фінальним дизайном — почитати актуальні джерела.

- iOS підтримує Web Push у PWA **тільки з iOS 16.4+** і **тільки коли додаток доданий на головний екран** (не у Safari-вкладці).
- Дозвіл на сповіщення можна запитати **лише у відповідь на жест користувача** (тап), не автоматично.
- Підписки на iOS періодично **відвалюються** (при оновленні додатку, зміні SW, тривалому простої) — треба вміти це детектувати і перепідписувати (частково вже є `pushKeysEqual` для ротації VAPID).
- `applicationServerKey` на iOS може бути недоступний для читання — обережно з логікою порівняння ключів (вже враховано в коді, не зламати).

**Що пошукати:** «iOS PWA web push reliability 2026», «web-push schedule future notification», поведінка `pushManager.subscribe` на iOS, ліміти.

## 5. Системне рішення (мета — чесні + надійні нагадування)

Принцип: **кнопка дзвіночка має показувати РЕАЛЬНИЙ стан, а нагадування — реально приходити для сьогодні І майбутніх днів.**

### 5.1. Зробити push для майбутніх днів
- Якщо бекенд це вже вміє (п.3) → **послабити клієнтський guard**: підписувати рейс одразу при збереженні, незалежно від дати. Сервер вистрелить у день рейсу.
- Якщо бекенд НЕ вміє → додати: підписувати при збереженні з `track_date` + cron-логіку, яка щодня бере підписки на «сьогодні» і шле T-15/T-0.

### 5.2. Чесний стан дзвіночка (3 стани замість 2)
Замість «вкл/викл» показувати реальність:
- 🔔 **Активне** — є дозвіл + підписка реально створена.
- 🔕 **Вимкнено користувачем** — `notify=false`.
- ⚠️ **Недоступно** — немає дозволу на сповіщення / не iOS-PWA / підписка впала. Тап → пояснення «Увімкніть сповіщення в налаштуваннях» або запит дозволу.

### 5.3. Зворотний зв'язок при збереженні
- Зберіг рейс → якщо push недоступний, **сказати одразу** (тост): «Збережено. Сповіщення вимкнені — увімкніть щоб отримати нагадування». Не мовчати.

### 5.4. Стійкість
- При відкритті Автобусів / поверненні в додаток — **звіряти** локальні `notify=true` рейси з реальними підписками в Supabase і перепідписувати втрачені (self-heal).
- Логувати збої гучно (не німий `catch`).

## 7. Критерій готовності (acceptance)

- Збережений рейс на **сьогодні І завтра** реально присилає push (перевірено на пристрої).
- Якщо сповіщення недоступні — дзвіночок це **показує** (⚠️), і користувача попереджено при збереженні.
- `notify=true` без реальної підписки **неможливий** (self-heal або чесний стан).
- Жодного німого `catch` у push-шляху.
