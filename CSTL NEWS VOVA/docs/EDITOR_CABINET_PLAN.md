## ✅ ЗАТВЕРДЖЕНА АРХІТЕКТУРА (рішення Вови, 04.07.2026)

**Не окрема `editor.html`, не старий email+пароль — а ЄДИНА адмін-панель з ролями (RBAC).**

**RBAC** (Role-Based Access Control — керування доступом на основі ролей/дозволів). Суть:

1. **Сховище контенту — Supabase** (Питання 1 → B). Новини/події переносяться з git-файлів у базу; форма пише в Supabase.
2. **Вхід — Google для ВСІХ** (Питання 2 → A). Ти, Роман, Алла і майбутні люди входять через Google. Старий email+пароль `admin.html` мігрує на Google.
3. **Одна спільна панель + система permissions** (Питання 3 → єдина панель). Не дублюємо адмінки/сторінки.
4. **Модерація = дозвіл** (Питання 4 → через permissions). «Публікує сама чи через схвалення» — це прапорець `news.publish`, а не окреме рішення.

### Модель даних (ескіз)
- `app_users` — `uid` (Google-ідентифікатор), ім'я, пошта, чи супер-адмін.
- Набір **permissions** (дозволів-прапорців): `news.create`, `news.edit`, `news.delete_own`, `news.publish`, `events.create`, `events.edit`, `board.moderate`, `board.spam`, `users.manage`, …
- **Супер-адміни** (Вова + Роман) — усі дозволи + право додавати людей і роздавати права.
- **Алла** — тільки `news.*` + `events.*`; НЕ бачить модерацію Дошки/скарги/антиспам/системне.

### Складові реалізації
1. Таблиця людей + дозволів у Supabase.
2. Екран керування людьми (лише для `users.manage`) — додати за Google-поштою + галочки розділів.
3. Панель показує лише дозволені розділи (кожен таб/кнопка перевіряє свій дозвіл).

### ⚠️ Критичні застереження
1. **Безпека — на рівні бази, не інтерфейсу.** Ховати кнопки недостатньо: справжня заборона = **RLS** (Row Level Security — правила Supabase на рівні рядків: писати в `articles` дозволено лише `uid` з дозволом `news.create`). Інтерфейс лише відображає дозволи, забороняє — база.
2. **Це переробка наявної `admin.html`** (email+пароль → Google + ролі), а не доповнення збоку. Модерація Дошки стає дозволом `board.moderate`. Обсяг більший за «просто кабінет».
3. **Вова + Роман — перші рядки в таблиці** (одноразово занести Google-`uid` як супер-адмінів, інакше після міграції ніхто не зайде).

### Питання 5 (відео) → рішення: НЕ інтегруємо в MVP
- Замість embed/файлів — **проста кнопка/посилання** на Instagram/YouTube Olyka Castle. Легше технічно + **воронка трафіку** з додатка в соцмережі (ростимо їхню аудиторію).
- Повноцінне відео (embed або власні файли) — пізніше, за реальної потреби, поза MVP.

### Питання 6 (чернетки + планування) → рішення: B (чернетки зараз, планування потім)
- **Чернетки (draft)** — закладаємо в MVP. Дешево: прапорець `status: draft` у базі, без таймера. Алла може зберегти недописане і дописати пізніше.
- **Планування на дату (scheduled)** — ПІЗНІШЕ. Потребує cron (таймер на сервері що сам публікує в заданий час) — окрема інфраструктура, поза MVP.

## Рішення (Рома, 01.07)

Будуємо **Кабінет редактора** (варіант B) — веб-форма публікації новин / статей / подій / історій для Алли.

**Контекст рішення:**
- Алла — контент-менеджерка проекту (відео, статті, медіа-робота). Події громади створюватиме переважно вона (див. `CONTENT_STRATEGY.md` → «Команда контенту»).
- Наявна адмінка `admin.html` — **тільки для Дошки громади** (модерація постів мешканців, оголошення, коментарі, адміни). Публікації новин/статей там НЕМА.
- Новини/події/історії зараз живуть у git-файлах (`data/articles.json`, `data/events.json`, `data/olyka-stories.json`) — редагувати їх з веб-форми без бекенду неможливо. Єдиний обхід — правка JSON через GitHub UI (технічно, не для Алли).
- Тому найчистіший шлях: **перенести редакційний контент у Supabase** (як Дошку) і дати Аллі форму. Деталі залежать від відповідей нижче.

**Відхилений варіант A (тимчасовий):** інструкція + правка JSON через GitHub — лишається запасним, поки кабінет не готовий.
