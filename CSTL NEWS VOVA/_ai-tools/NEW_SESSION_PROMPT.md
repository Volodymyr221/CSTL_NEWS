## Ключові правила які ти ОБОВ'ЯЗКОВО маєш виконувати

### 🔴 Про зміни в коді
- **БЕЗ "Роби" — код не чіпати.** Я скажу явне "Роби" — і тільки тоді ти щось міняєш. До цього — тільки обговорення, питання, пропозиції, план.
- **Один логічний крок за раз.** Не роби 10 задач одним ходом. Один хід Claude = один логічний блок роботи = один коміт.
- **Читай код перед змінами.** Ніколи "на пам'ять". Відкрий потрібний файл через Read tool, прочитай, тільки тоді пропонуй.
- **Не губи попередні фікси.** При кожній зміні перевіряй що попередні виправлення збережені.
- **Перевіряй синтаксис** після JS змін через `node --check файл.js`.

### 🟡 Про комунікацію
- **Пояснення англомовних термінів у дужках.** Після кожного англійського слова — одразу у дужках пояснення українською. Я вчусь програмуванню паралельно через цей проект. Без пояснень я не розумію. Приклади:
  - `PWA` (Progressive Web App — веб-додаток що встановлюється з браузера)
  - `commit` (збережена зміна в git)
  - `deploy` (публікація сайту)
  - `bundle.js` (один файл зі склеєними модулями JavaScript)
  - `Service Worker` (фоновий скрипт який керує кешуванням PWA)
- **Короткі повідомлення.** Я комунікую коротко, очікую того самого. Одна думка за раз.
- **Скріншоти** — мій основний спосіб показати баг або ідею.
- **Сперечайся чесно.** Якщо бачиш слабке місце в моїй ідеї — кажи. Не погоджуйся заради ввічливості.
- **Пропонуй кращі рішення.** Ти краще знаєш технічні можливості — активно пропонуй альтернативи, навіть якщо я не питав.
- **Пояснюй свої дії.** Після кожної технічної дії — коротко що зробив, для чого, як працює.

### 🟢 Про автоматизацію і workflow
- **Автокоміт через `~/.claude/stop-hook-git-check.sh` — це фіча, не баг.** Не переписуй хук. Він страхує роботу на випадок ліміту API і дозволяє мені паралельно тестувати на живому сайті.
- **Двогілковий потік (branching workflow):**
  - **Робоча гілка** — `claude/start-session-XXX` (створюється автоматично при старті сесії). Тут йде вся розробка, експерименти, недороблене.
  - **Production (бойова) — `main`** — тільки готові узгоджені зміни. Мерджимо через скіл `/finish` (PR → squash merge → автодеплой).
  - **Не пушити в `main` напряму** з робочої сесії — тільки через `/finish` коли я сказав "готово".
- **Лічильник версії внизу екрану** (формат `v{N} · DD.MM HH:MM`) — мій індикатор що деплой пройшов. Якщо після `/finish` число не змінилось → деплой не пройшов.
- **При деплої КОДУ** (зміни у `src/`, `index.html`, `style.css`, `sw.js`) — міняй `CACHE_NAME` у `sw.js` (формат `cstl-YYYYMMDD-HHMM`). При зміні тільки `.md` файлів — **не чіпай**.
- **`bundle.js` комічу в git** (варіант Б, робоча база). CI перегенерує свіжий при деплої, але не комітить назад.
- **Після КОЖНОЇ зміни** оновлюй `_ai-tools/SESSION_STATE.md`.

## Чому цей промпт стабільний

- Конкретні баги, задачі, номери версій — **не в промпті**. Вони у `BACKLOG.md`, `CSTL_BUGS.md`, `SESSION_STATE.md`, які Claude читає при старті.
- Правила роботи, стиль комунікації, профіль Вови — **змінюються рідко**. Якщо і змінюються — оновлення цього файлу разове.
- Формат відповіді — **стандартний шаблон**. Новий Claude кожного разу дає однакову за структурою відповідь, не плутає мене.

Тому цей файл треба оновлювати **тільки коли:**
- Змінились правила роботи (нові заборони, нові дозволи)
- Змінилась фундаментальна структура проекту (додано важливий файл у обов'язкове читання)
- Змінився стиль комунікації Вови

**НЕ оновлювати** на кожен закритий баг, кожну нову задачу, кожен коміт. Це не роль цього файлу.

## Як запускається скіл `/startuem`

У корені проекту є файл `.claude/commands/startuem.md` — він каже Claude "прочитай `NEW_SESSION_PROMPT.md` і виконай те що там". Тобто:

```
Вова у новому чаті пише: /startuem
    ↓
Claude Code завантажує .claude/commands/startuem.md
    ↓
startuem.md каже: "Прочитай CSTL NEWS VOVA/_ai-tools/NEW_SESSION_PROMPT.md"
    ↓
Claude читає цей файл і виконує інструкції всередині
    ↓
Claude читає всі 6 обов'язкових файлів, формулює відповідь
```

Існуючий глобальний скіл `/start` (для інших проектів Вови) **не чіпаємо** — для CSTL NEWS є окремий скіл `/startuem`, щоб не було конфлікту.
