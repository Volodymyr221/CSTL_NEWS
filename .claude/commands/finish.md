# /finish — Завершити сесію і опублікувати на сайті (мердж робочої гілки в main)

> **Замінює старий `/deploy`.** Робить пуш робочої гілки → створює PR → мерджить у `main` → запускає автодеплой через GitHub Actions.

Виконуй кроки по порядку. Не пропускати жодного.

---

## 🔴 Крок 0 — Живий журнал сесії (перевірка перед фінішем)

> **Правило (діє ВСЮ сесію, не лише на фініші):** у КОЖНІЙ сесії ведеться живий журнал
> `_session-log/vova-<РРРР-ММ-ДД>.md` (напр. `_session-log/vova-2026-07-23.md`).
> Його СТВОРЮЮТЬ на старті (`/startuem` Крок 0) і дописують **по ходу — після кожної значущої дії**, а НЕ наприкінці з пам'яті.
> Мета: підсумок для brain (Крок 8) має спиратися на **записаний факт**, а не на реконструкцію з пам'яті великої сесії (контекст губиться → звіт виходить неповним).

**Що робити на фініші:**
1. Переконайся що журнал сесії існує і містить усі значущі дії цієї сесії. Якщо його немає або він відстає — **допиши зараз усе що пам'ятаєш** (файли, PR, деплої, рішення, інциденти, глухі кути).
2. Кожен рядок журналу фіксує: **що зроблено** (файл/функція) · **навіщо** · **що пробував і що НЕ спрацювало** · **рішення/слова власника дослівно** · **інцидент і як виправив** · **PR#** · **деплой#** · **CACHE_NAME**.
3. Журнал комітиться разом зі змінами (це `_session-log/*.md` — на CACHE не впливає).
4. Крок 8 (підсумок для brain) генеруй **із журналу**, а не з пам'яті.

---

## Крок 1 — Час
Запусти `date` в терміналі. Запам'ятай результат — він потрібен для CACHE_NAME (назви кешу).

## Крок 2 — CACHE_NAME (версія збережених файлів у Service Worker)
Відкрий `sw.js` і зміни `CACHE_NAME` на формат `cstl-YYYYMMDD-HHMM`.
Приклад: `cstl-20260430-1430`. НЕ вигадувати час — використовувати результат з Кроку 1.

**Виняток:** якщо змінювались **тільки** `.md` файли або `.claude/` команди — `CACHE_NAME` НЕ чіпати.

## Крок 3 — Перевірка синтаксису (node --check)
Для кожного зміненого `.js` файлу:
```
node --check шлях/до/файла.js
```
Якщо помилка — СТОП. Виправити, тільки тоді далі.

## Крок 4 — Оновлення документації

### ЗАВЖДИ — `CSTL NEWS VOVA/_ai-tools/SESSION_STATE_VOVA.md`
- **Оновлено:** сьогоднішня дата
- **Що зроблено:** коротко що змінювалось у сесії
- **Відкриті задачі:** оновити якщо щось закрито або з'явилось нове

### Якщо змінювалась структура файлів — `CSTL NEWS VOVA/CLAUDE.md`
Таблиця "Файлова структура" — додати новий файл.

### Якщо змінювалась архітектура — `CSTL NEWS VOVA/docs/ARCHITECTURE.md`
Оновити діаграми і описи.

## Крок 5 — Визначити поточну гілку
```
git branch --show-current
```

- Якщо `main` → перейди до Кроку 6А (старий прямий потік).
- Якщо інша (наприклад `claude/start-session-XXX`) → перейди до Кроку 6Б (потік через PR).

## Крок 6А — Прямий пуш у main (тільки якщо гілка вже main)
```
git add <конкретні файли — НЕ "git add ."
git -c commit.gpgsign=false commit -m "опис що зроблено"
git push -u origin main
```
Перейти до Кроку 7.

---

## Крок 6Б — Пуш робочої гілки + PR + merge

### 6Б.0 — Rebase на свіжий main (проти конфліктів і затирання)
> Гілка могла відстати від `main` (парсери новин/автобусів пушать автоматично). Синхронізуй ПЕРЕД PR,
> щоб мердж не тягнув старий код.
```
git fetch origin main
git merge origin/main --no-edit    # або git rebase origin/main; конфлікти вирішити ВРУЧНУ (ніколи -X theirs)
node --check <змінені .js>          # переконайся що після злиття код валідний
```

### 6Б.1 — Коміт на робочу гілку
```
git add <конкретні файли>
git -c commit.gpgsign=false commit -m "опис що зроблено"
git push -u origin <назва-поточної-гілки>
```

### 6Б.2 — Створити Pull Request (запит на злиття гілки в main)
Виклик GitHub MCP:
```
mcp__github__create_pull_request
  owner: Volodymyr221
  repo: cstl_news
  base: main
  head: <назва-поточної-гілки>
  title: <коротко що зроблено>
  body: <маркований список ключових змін>
```
Запам'ятати `number` (номер PR) з відповіді.

### 6Б.3 — Merge PR (злиття у main)
```
mcp__github__merge_pull_request
  owner: Volodymyr221
  repo: cstl_news
  pullNumber: <з 6Б.2>
  merge_method: squash
```
Після мерджу — автодеплой стартує на `main`.

## Крок 7 — Перевірка CI (continuous integration — автоматичні тести і деплой)
Через ~3 хв сайт має оновитись. Лічильник версії внизу екрану (`v{N} · DD.MM HH:MM`) має змінити число.

Якщо деплой не запустився — порожній коміт як ретригер:
```
git checkout main
git pull origin main
git -c commit.gpgsign=false commit --allow-empty -m "ci: retrigger"
git push origin main
git checkout <робоча-гілка>
```

## Крок 8 — 🧠 ПІДСУМОК ДЛЯ BRAIN (зовнішній контролер) — ОБОВ'ЯЗКОВО

> Вова веде окремий «другий мозок» (інший застосунок/репозиторій, цей Claude його НЕ бачить).
> Туди вставляється підсумок кожної сесії; brain-асистент архівує і дає кросс-сесійний контроль.
> **Наприкінці КОЖНОГО `/finish` (після оновлення стану) — згенеруй підсумок.**
> **🔴 ВИДАВАЙ ЗАВЖДИ ОДНИМ суцільним блоком — усе всередині одного код-фенсу ``` ``` ```, без розбивки на окремі повідомлення/жирний текст поза блоком — щоб скопіювати за ОДИН раз.**

**🧑 Імʼя у підсумку — Вовчик** (Володимир Шевчук, власник проєкту): пиши «сесія Вовчика» і `🗣️ Рішення Вовчика:`.

**🔴 Джерело = ЖИВИЙ ЖУРНАЛ сесії** (`_session-log/<роль>-<дата>.md`, Крок 0) + git-історія цієї сесії. **НЕ реконструюй із пам'яті** — контекст великої сесії губиться, і підсумок виходить неповним. Пройдись журналом від початку до кінця сесії.

**🔴 ВИЧЕРПНІСТЬ важливіша за стислість.** Підсумок читає аналітик-brain, не людина — **довжина НЕ обмежена, краще зайва деталь ніж втрачена**. НЕ пиши «основне» — пиши **ВСЕ**:
- **КОЖЕН** змінений файл, функція, PR (№), деплой (№), CACHE_NAME — поіменно, не «та інші».
- **КОЖНЕ** рішення + **ЧОМУ** саме так (не лише «що»).
- Усе що **обговорювали**, навіть те що НЕ втілили (ідеї, макети, напрямки).
- Що **НЕ спрацювало** / глухі кути / що пробував і відкинув — і **чому** (щоб не повторювати).
- **Інциденти** + як виправлено, з деталями (збої деплою, git-конфлікти, розсинхрони, відкати).
- **Хронологію по кроках** від початку до кінця сесії.
- Дослівні **слова/рішення власника** де є.

**Правила тексту:** так, щоб людина без доступу до коду зрозуміла; розшифровуй скорочення, не лишай внутрішніх відсилань без пояснення. Простою мовою — але **повно, нічого не стискаючи**.

**Формат (заповни реальними даними сесії):**

```
Підсумок сесії CSTL Life [РРРР-ММ-ДД] — сесія Вовчика (гілка [claude/startuem-* або vova/*]):

🗂️ Зона: над чим працював цю сесію (Автобуси/Дошка/Події/Новини/Push/Системне/Громада/Стрічка…).
🔧 Що зроблено (код/файли): конкретні зміни — файли, фічі, фікси.
💬 Що обговорили (навіть якщо не робили): ідеї, напрямки, дослідження.
🎯 Ключові рішення + ЧОМУ: технічні/архітектурні рішення сесії і причина кожного (не лише «що», а «чому саме так»).
❌ Що не спрацювало: підходи що спробували і відкинули (щоб не повторювати).
🗣️ Рішення Вовчика: що вирішив (прямі цитати корисні).
⏸️ Відкладене: що свідомо лишили на потім.
🔄 Конфлікти/суперечності: де думки розійшлись, де відкотили, що змінилось у плані.
⚠️ Інциденти: збої деплою, відкати, зламані білди, git reset/force (чесно).
👁️ Спостереження (стиль роботи цю сесію): як власник працював.
🔓 Відкриті питання: невирішене що чекає рішення.
➡️ Наступний крок: що логічно робити далі.
📊 Версія/кеш (якщо є): діапазон версій збірки.
🌿 Гілка + 🔗 коміти: гілка і діапазон комітів (від→до, к-сть).
```

В кінці блоку додай рядок: **«Запиши у brain.»**

> Контекст проекту (щоб підсумок був самодостатній): CSTL Life — PWA-центр життя
> громади міста Олика (новини + події + спільнота). Власник — Вовчик (Володимир
> Шевчук), проєкт одноосібний. Рів проекту = жива локальна
> громада + гіперлокальні дані + мережа; монетизація = % з потоку грошей
> (квитки на події/локальна комерція), не реклама.

## Що робить цей скіл — коротко

```
Робоча гілка (claude/start-session-XXX)
    ↓ git push
GitHub: гілка з готовими змінами
    ↓ create PR + merge PR (squash)
main отримує мердж-коміт
    ↓ deploy.yml запускається
GitHub Pages оновлюється через 1-3 хв
```

**Принцип:**
- `main` — production (бойова): тільки готові узгоджені зміни.
- Робоча гілка — експерименти, недороблене, обговорення.
- `/finish` = я (Вова) сказав "готово, публікуй".
