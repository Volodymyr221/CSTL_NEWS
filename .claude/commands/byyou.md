# /byyou — напівавтономний режим (CSTL)

## Що це
Claude отримує ціль, планує 10-15 кроків і виконує потоком, зупиняючись лише на контрольних точках. Власник (Вова) контролює результат, не кожен крок. Скоуп: CSTL. Стан потоку — у CSTL NEWS VOVA/_ai-tools/BYYOU_PLAN.md

## Коли
Великі цілі (≥3 файли або ≥1 год). НЕ для <30 хв (звичайний діалог або /fix).

## Команди
/byyou <ціль> — нова задача (план → «ок» → виконання). /byyou — продовжити незавершений потік з BYYOU_PLAN.md

## Режим глибокої роботи (перед Фазою 0)
Думай широко. Копай глибоко. Використовуй агентів де треба. Будуй системні рішення. Помічай неочевидне. Не дій наосліп. Перевіряй. Не будь наївним. Дивись наперед.

## Дві брами
1. Брама старту — показати план (10-15 кроків) → чекати «ок» власника → тоді виконувати.
2. Брама деплою — між брамами лише локальні коміти, БЕЗ push. Публікація тільки коли власник каже слово «деплой». Хук byyou-push-lock.js блокує push поки план активний без цього слова.

## Стани кроків
🟢 GO — виконую (читання/код/тест/локальний коміт). 🟡 ТВОЯ ЧЕРГА — треба смоук на iPhone або ввід власника. 🔴 СТОП — питаю спершу: архітектура, зміна схеми Supabase, нові ключі сховища, незворотні дії, конфлікт з RODMAP.

## Видимість
Кожне повідомлення з [/byyou N/M]. Перед кроком — одне речення. Після — що зроблено + хеш коміту + наступний крок.

## Сторож контексту 75%
Хук byyou-context-guard.sh блокує тихий вихід при ≥75%: дозаповнити BYYOU_PLAN.md «Де зупинились», Статус: paused, гучне повідомлення власнику. Новий чат → /byyou → продовжить.

## Фаза 0: Вхід
1. Прочитати «Режим глибокої роботи».
2. git branch --show-current — працюю на гілці сесії (`claude/startuem-...` або `vova/*`), НЕ напряму в main. Глянути останні коміти.
3. Якщо BYYOU_PLAN.md зі статусом active/paused → режим ПРОДОВЖЕННЯ (прочитати весь файл, Фаза 2). Інакше → НОВИЙ (прочитати SESSION_STATE_VOVA.md + RODMAP.md розділ Active + AUDIT_2026-07.md).

## Фаза 1: Планування (тільки НОВИЙ)
1. Прочитати РЕАЛЬНИЙ код усіх задіяних файлів (не з памʼяті).
2. Нова фіча → короткий ресерч (WebSearch аналоги). Баг → 3 питання (корінь / межі / DRY).
3. Принципи: найпростіше = найкраще; корінь, не симптом; мінімум у фічах, максимум у фундаменті.
4. Скласти 10-15 кроків у BYYOU_PLAN.md (таблиця, кожен 🟢/🟡/🔴). Кроки що чіпають архітектуру/схему/сховище → одразу 🔴.
5. Поставити Статус: active (вмикає push-замок).
6. Оцінка обсягу: N кроків × ~35K токенів ≈ X% вікна. Якщо >60% — запропонувати розбити на 2 потоки.
→ БРАМА СТАРТУ: показати план (≤20 рядків) + оцінку → «Стартуємо? (так / зміни крок N / розбити / скасувати)». Чекати «ок».

## Фаза 2: Потік (цикл на крок)
1. Прочитати код у точці зміни (файл міг змінитись).
2. Виконати: файли >250 рядків → каркас + Edit частинами; зміни в src/* → підняти CACHE_NAME у sw.js (формат cstl-YYYYMMDD-HHMM).
3. Перевірити: node --check <файл>.
4. Локальний коміт (БЕЗ push): feat(<зона>): крок N — <опис> [byyou]
5. Оновити BYYOU_PLAN.md: крок ✅ + хеш + «Де зупинились».
6. 🔴/🟡 крок → стоп, спитати, чекати. Дві невдалі спроби фіксу поспіль → пре-мортем → стоп (не латка-на-латку).
Пульс: 🟢 [4/11] Парсер дати · abc1234 · далі: 5 — фільтр

## Фаза 3: Тест-блок
1. node --check усіх змінених .js.
2. Браузерний смоук: /qa-explore (рівень за обсягом блоку) — збирає застосунок локально, ганяє Chromium по зачеплених екранах, ловить падіння + помилки консолі + скріншоти. Знайдене фіксити ДО реліз-нотаток. Якщо браузер недоступний — fail-soft (node --check + ручний iPhone).
3. /audit (перечитати змінене, баланс div, дублі, мертвий код).
4. Заповнити Реліз-нотатки у BYYOU_PLAN.md: ЩО ЗМІНИЛОСЬ / ЩО МОЖЕ ЗЛАМАТИСЬ / ЩО ПЕРЕВІРИТИ НА IPHONE (≤5, лише видимі фічі).
5. Показати реліз-нотатки власнику.

## Фаза 4: Брама деплою
1. Перед публікацією: node --check усіх змінених .js + браузер-смоук зелений. Червоне = фіксити, не деплоїти.
2. Сказати: «Блок готовий. Реліз-нотатки вище. Скажи «деплой» — публікую.»
3. Після «деплой»: підняти CACHE_NAME (якщо src змінювався) → деплой як /finish (гілка сесії → PR через GitHub MCP → squash-merge у main → deploy.yml деплоїть на Pages ~3 хв).
4. Перевірити версію-лічильник унизу екрана (v{N} · DD.MM HH:MM).
(У CSTL публікація — на злитті в main, не на push гілки; тому «деплой» = дозвіл і на push, і на злиття.)

## Фаза 5: Вихід / звіт
1. Підсумок власнику: зроблено (кроки + хеші), що відкрито, наступний крок.
2. Статус: done → перенести план в архів у _ai-tools/, скинути live-файл у шаблон idle.
3. Викликати /finish (оновити SESSION_STATE / RODMAP).

## Вихід + стоп-слова
«стоп» (вихід/пауза/досить) → стоп, Статус: paused, робота збережена: «🏁 Вийшов з /byyou на кроці N. Збережено, /byyou продовжить.» Авто-вихід — план виконано (Фаза 5). «відкат»/abort → стоп + запропонувати git reset --hard byyou-start-<сесія> (тільки з явним «ок»; якщо нічого не запушено — безпечно).

## Ніколи в автономії без дозволу
Немає git push без «деплой» · немає git push --force · немає git reset --hard (крім відкату з «ок») · немає злиття в main без «деплой» · немає нового ключа сховища / зміни схеми Supabase без 🔴-стопу · немає змін правил/CLAUDE.md.
